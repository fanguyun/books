# 场景模式

## 1. 想知道“为什么一切最后都是 0 和 1”

**适用场景**
- 初学者对照片、音乐、程序为什么都能变成二进制感到抽象。

**判断顺序**
1. 先解释数字系统为什么需要有限状态。
2. 再解释编码如何把文本、数字、图像映射成位序列。
3. 最后解释硬件如何存储这些状态。

**使用方式**
- 从 `模拟 vs 数字 -> 位/字节 -> 编码 -> 存储介质` 这条链路讲。

---

## 2. 想搞懂 CPU、内存和硬盘的分工

**适用场景**
- 初学者常把存储、内存和处理器混成一团。

**判断顺序**
1. 谁负责执行？
2. 谁负责临时保存运行中的数据？
3. 谁负责长期保存？

**使用方式**
- 先讲 CPU / RAM / 外部存储三角，再补程序加载与执行流程。

---

## 3. 想理解程序是怎么跑起来的

**适用场景**
- 问“写的代码为什么能变成软件”。

**判断顺序**
1. 源代码用什么语言写？
2. 编译/解释后变成什么？
3. CPU 最终执行的是哪一层？

**使用方式**
- 顺着 `源代码 -> 编译/汇编 -> 可执行文件 -> 机器码 -> CPU` 讲。

---

## 4. 想把某个故障定位到正确层次

**适用场景**
- 系统出问题时，不知道是硬件、OS、网络还是应用问题。

**判断顺序**
1. 是不是硬件或资源不可用？
2. 操作系统有没有提供正确抽象和权限？
3. 应用层逻辑是否正确？
4. 是否涉及网络与协议链路？

**使用方式**
- 先分层，避免把所有问题都扔给“程序有 bug”。

---

## 5. 想解释访问网站时发生了什么

**适用场景**
- 想从浏览器输入 URL 讲到页面显示。

**判断顺序**
1. URL 是什么。
2. 浏览器如何发 HTTP 请求。
3. 请求如何通过 TCP/IP 发送。
4. 服务器如何响应。
5. 浏览器如何渲染页面。

**使用方式**
- 用 `URL -> DNS/连接 -> HTTP -> 响应 -> 渲染` 的顺序讲。

---

## 6. 想设计学习路线而不是死记术语

**适用场景**
- 初学者想系统学计算机原理。

**判断顺序**
1. 先打数据表示基础。
2. 再学电路与硬件执行。
3. 再进软件、OS、网络、Web。

**使用方式**
- 按“向下扎根、向上抽象”的节奏，不要一开始跳到云计算和框架。
